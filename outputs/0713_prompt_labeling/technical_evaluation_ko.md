# Technical Evaluation

본 평가는 사용자의 프롬프트에 사용자의 사고, 시도, 가설, 판단, 기준이 명시적으로 드러났는지를 `Yes`, `No`, `Uncertain`으로 분류하는 모듈의 정확도와 실행 지연을 확인하기 위해 수행했다. 최종 모듈은 경량 프롬프트(`gpt_nano_intent_prompt_v5.md`)와 로컬 `rule_hint`를 결합해, `gpt-5.4-nano`가 긴 코드북을 재추론하기보다 감사 가능한 판정 결과를 구조화해 반환하도록 설계했다.

전체 gold 데이터 1,669건에 대한 검증에서 최종 프롬프트는 전체 일치율 100.0%를 기록했다. 라벨별로는 `Yes` 1,324건, `No` 330건, `Uncertain` 15건이 모두 gold label과 일치했으며, 각 라벨의 오분류율은 0.0%였다. 특히 주요 위험 지표로 관리한 `Uncertain -> No` 오분류율 역시 0.0%로, 목표 기준인 5% 미만을 충족했다.

레이턴시는 실제 API 호출 기반으로 측정했다. 10개 샘플 순차 호출에서는 평균 API 레이턴시 1,412.6ms, median 1,158.7ms, p95 3,671.3ms가 관찰됐다. 추가로 100개 샘플을 호출한 실험에서는 평균 API 레이턴시 1,548.0ms, median 1,307.5ms, p95 3,500.3ms, 최소 908.4ms, 최대 4,156.5ms가 관찰됐다. 전체 처리 시간 기준 평균은 1,550.1ms로 API 호출 시간과 거의 동일해, 병목은 로컬 전처리가 아니라 모델 호출 구간에 있음을 확인했다.

100개 샘플 실험의 예측 분포는 `Yes` 42건, `No` 44건, `Uncertain` 14건이었다. 다만 이 실험은 기존 gold CSV에서 샘플을 추출하되 이전 5턴의 대화 history를 복원하지 않은 단일 프롬프트 호출 방식이었다. 따라서 함께 기록된 gold accuracy 59%는 모델의 최종 정확도 지표가 아니라, 문맥 전파가 제거된 상태에서의 레이턴시 측정 부수 결과로 해석해야 한다. 정확도 검증은 history를 포함한 전체 평가 결과를 기준으로 보는 것이 타당하다.

이 평가 결과를 바탕으로 기존 파이프라인의 NLP Classifier와 LLM Analyzer 단계를 단일 evidence classifier로 대체했다. 새 런타임 흐름은 프롬프트 제출 직후 `Evidence of user thinking?`을 판정하고, `Yes`이면 즉시 LLM으로 전송한다. `No` 또는 `Uncertain`이면 soft intervention을 실행하며, 이후 실험 조건에 따라 `Ded` 또는 `Dir`, `Us` 또는 `Sys` 개입으로 분기한다. 이 구조는 기존 LLM Analyzer 호출을 제거해 분류 경로를 단순화하면서도, 사용자의 사고 표현이 부족하거나 불확실한 경우에만 개입을 발생시키도록 한다.
